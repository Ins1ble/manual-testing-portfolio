# Тест-план

## Объект тестирования

Интернет-магазин «ShopExample».  
Включает каталог товаров, корзину, оформление заказа, оплату, личный кабинет.

## Цели

*   Убедиться в корректной работе функциональности корзины и оформления заказа
*   Проверить устойчивость системы к некорректным данным
*   Оценить интеграцию с платёжным шлюзом (имитация)
*   Проанализировать удобство пользовательского пути

## Стратегия

*   Уровни тестирования: компонентное (UI), интеграционное (API)
*   Виды тестирования: функциональное, регрессионное, smoke
*   Техники: чек-листы для основного потока, тест-кейсы для критичных сценариев, исследовательские сессии

## Учёт модели разработки

Проект ведётся по Agile (Scrum). Тестирование начинается внутри спринта, сразу после готовности первой версии функциональности.  
При необходимости возможен переход к Waterfall для финальной приёмки — это отражено в критериях начала и окончания.

## Критерии начала

*   Развёрнуто тестовое окружение (сервер, БД, фронтенд)
*   Доступна сборка с описанными изменениями
*   Подготовлены тестовые данные (пользователи, товары, промокоды)

## Критерии окончания

*   Все критические и блокирующие дефекты закрыты
*   Пройдено 100% проверок по чек-листу
*   Покрытие требований тест-кейсами не менее 90%
*   Отсутствуют открытые баги с приоритетом High и выше

## Риски

*   Нестабильность окружения замедляет проверки
*   Отсутствие документации на API увеличивает время исследования
*   Ограниченный набор промокодов снижает покрытие

## Расписание (оценка)

*   Анализ требований: 2 часа
*   Разработка чек-листов и тест-кейсов: 4 часа
*   Выполнение тестов: 8 часов
*   Документирование дефектов: 2 часа
*   Подготовка отчёта: 1 час
